核心能力

页面更现代，内容也更贴近真实 .NET 使用场景。

Senlinz.Localization 重点解决团队真正关心的问题：字符串维护、生成代码可读性，以及简单稳定的运行时模型。

生成的 API

把 JSON 键直接变成可发现的 C# 成员。

普通文本生成属性，带占位符的文本生成方法，命名与源文件保持一致，IDE 中也更容易查找和补全。

  • hello → L.Hello
  • sayHelloTo → L.SayHelloTo(string name)
  • 生成类型保持简洁，适合日常维护与协作
资源体系

只覆写真正变化的语言文本。

每种语言一个资源类即可，默认文本继续由生成器维护，减少重复实现。

回退机制

缺少语言或键时仍能稳定输出。

当资源不存在或未覆写某个值时，会自动回退到 JSON 中的默认文本。

枚举本地化

让枚举值自然接入本地化流程。

通过特性把枚举成员映射到本地化键，不需要手写额外转换层。

运行时抽象

保持轻量，按需引入。

若只需要共享契约，可单独使用 Senlinz.Localization.Abstractions。

快速开始

从 JSON 到本地化输出，只需要几个清晰步骤。

默认使用 l.json，也可以通过项目属性自定义文件名。

  1. 创建本地化文件

    先定义好将要生成 API 的源文本。

    {
  "hello": "Hello",
  "sayHelloTo": "Hello {name}!",
  "statusReady": "Ready"
}

  2. 在项目中注册

    让源码生成器能够读取该文件，并在需要时复制到输出目录。

    <ItemGroup>
  <AdditionalFiles Include="l.json" />
  <None Update="l.json" CopyToOutputDirectory="PreserveNewest" />
</ItemGroup>

  3. 在运行时解析文本

    直接传入资源类，或接入已有资源提供器。

    var resolver = new LStringResolver(
    () => currentCulture,
    new EnResource(),
    new ZhResource());

Console.WriteLine(resolver[L.Hello]);
生成类型

框架会为你准备什么

  • L：强类型访问器
  • LResource：生成的资源覆写基类
  • LString：包含键、默认文本与参数
  • LStringResolver 与 LResourceProvider：运行时解析入口
适用场景

适合哪些团队

如果你希望本地化键与代码更贴近、喜欢编译期即可发现 API，并且不想引入重量级本地化框架，这个方案会很合适。

  4. 最终生成的 .nupkg 与可选 .snupkg 会被发布到 NuGet。
